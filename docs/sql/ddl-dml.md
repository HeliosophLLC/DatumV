title: DDL / DML
---

DatumIngest supports both persistent and session-scoped tables with DDL/DML routed through `TableCatalog`. Persistent tables materialise as `.datum` files alongside the catalog's `.datum-catalog.json`; temp tables live in-memory and die with the catalog.

### Table mutability

Every table registered in the catalog has a mutability level:

| Level | Description |
|-------|-------------|
| `ReadOnly` | Default for all data sources (CSV, Parquet, JSON, system tables). DDL/DML statements return an error. |
| `SessionOwned` | Temp tables created within a session. DDL/DML permitted. |
| `Writable` | Persistent `.datum` tables created via `CREATE TABLE`. DDL/DML permitted. |

`ITableProvider` exposes three opt-in flags â€” `CanAlterColumns`, `CanAppendRows`, `CanDeleteRows` â€” gating the corresponding mutation paths. System tables (information_schema, datum_catalog.\*, models, udfs, â€¦) leave the defaults at `false`, so the catalog rejects mutations against them with a clear `"Table 'X' is read-only"` error.

### CREATE TABLE

Creates a persistent table backed by a `.datum` file in the catalog directory:

```sql
CREATE TABLE features (
    customer_id   Int32 PRIMARY KEY,
    tenure_months Int32 NOT NULL,
    monthly_spend Float64,
    label         String
)
```

Persistent `CREATE TABLE` requires a catalog backed by `.datum-catalog.json` (i.e. opened with a catalog path). Without one, use `CREATE TEMP TABLE` instead.

`CREATE TABLE IF NOT EXISTS` silently succeeds when a table with the same name already exists; the existing table's schema is not validated against the new declaration.

#### Type names

Column types use the kind names from the type system: `Int8`, `Int16`, `Int32`, `Int64`, `UInt8`, `UInt16`, `UInt32`, `UInt64`, `Float32`, `Float64`, `Boolean`, `String`, `Uuid`, `Date`, `DateTime`, `Time`, `Duration`. See [Type System](type-system.md) for the full set.

#### Column modifiers

| Modifier | Behavior |
|----------|----------|
| `NOT NULL` | Column rejects NULL values on `INSERT`. |
| `PRIMARY KEY` | Implies `NOT NULL`. Enforces uniqueness on `INSERT` â€” duplicate key values are rejected with `PrimaryKeyViolationException`. PK columns must be fixed-size kinds (Int*, UInt*, Float*, Boolean, Date/Time/DateTime/Duration, Uuid); the total key size must be â‰¤ 16 bytes. |
| `DEFAULT <expr>` | Column receives the value of `<expr>` when omitted from an `INSERT`. Any tableless expression is accepted â€” literals, arithmetic (`1 + 2`), string concatenation (`'a' \|\| 'b'`), function calls (`now()`, `uuidv4()`), array literals (`[1, 2, 3]`), `CASE` expressions, casts. The expression is evaluated per row, so `DEFAULT now()` captures a fresh timestamp for each row and `DEFAULT uuidv4()` produces a distinct UUID per row. Column references, subqueries, and window functions are rejected â€” there's no source row in scope at `DEFAULT`-evaluation time. |
| `IDENTITY` / `IDENTITY(seed, step)` | T-SQL-flavored alias for `GENERATED ALWAYS AS IDENTITY`. Auto-generates an integer value on each `INSERT`; explicit values are rejected. Bare form defaults to `seed=1, step=1`. Step may be negative. At most one `IDENTITY` column per table. The column kind must be a signed or unsigned integer (Int8/16/32/64, UInt8/16/32/64); 128-bit integers are not supported. |
| `GENERATED ALWAYS AS IDENTITY` / `GENERATED ALWAYS AS IDENTITY(seed, step)` | PG-canonical syntax for `IDENTITY`. Same semantics as bare `IDENTITY` â€” explicit values rejected on `INSERT`. |
| `GENERATED BY DEFAULT AS IDENTITY` / `GENERATED BY DEFAULT AS IDENTITY(seed, step)` | PostgreSQL "user wins" variant. Explicit values supplied by an `INSERT` are accepted as-is; the counter is consulted only when the column is omitted from the column list. The counter is **not** auto-advanced by explicit inserts â€” if you explicitly insert a value the counter would later generate, a subsequent omitted `INSERT` will hit the collision (and fail if a `PRIMARY KEY` is involved). |
| `AS (expr)` / `GENERATED ALWAYS AS (expr)` | `STORED` computed column. The value is materialised per row from `expr` at `INSERT` time; explicit values are rejected on both `INSERT` and `UPDATE`. The expression references other columns of the same row. Mutually exclusive with `DEFAULT` and `IDENTITY`. STORED only â€” `VIRTUAL` is not yet supported. See [Computed columns](#computed-columns) below for the full surface. The bare `AS (expr)` form is a shorthand for `GENERATED ALWAYS AS (expr)`. |

Column-modifier order in the parser is fixed: `NOT NULL â†’ PRIMARY KEY â†’ DEFAULT â†’ GENERATED-or-AS-or-IDENTITY`. `Int64 IDENTITY PRIMARY KEY` parse-fails; write `Int64 PRIMARY KEY IDENTITY` or the PG-canonical `Int64 PRIMARY KEY GENERATED ALWAYS AS IDENTITY`.

Composite primary keys are declared with a table-level constraint:

```sql
CREATE TABLE order_products (
    user_id    Int32,
    product_id Int32,
    quantity   Int32,
    PRIMARY KEY (user_id, product_id)
)
```

#### PRIMARY KEY enforcement

Single-column `PRIMARY KEY` is backed by a maintained mutable B+Tree in a `.datum-pkindex` sidecar â€” uniqueness checks probe the tree in `O(log table_size)` per row. Composite `PRIMARY KEY` falls back to a scan-based check (loads existing PK values into a `HashSet` at `INSERT` start). Both paths reject within-batch duplicates and `NULL` in any PK column.

#### IF NOT EXISTS

`CREATE TABLE IF NOT EXISTS` does a name-only existence check; no schema-equivalence comparison. If a table already exists, the statement is a no-op regardless of whether the requested schema matches.

#### AT 'path'

`CREATE TABLE â€¦ AT 'path'` lets the caller place the backing `.datum` file at a specific location instead of the catalog directory. This clause is rejected by default â€” production catalogs don't honor it. Pass `allowExplicitTablePaths: true` to the `TableCatalog` constructor to opt in (typically test fixtures only).

### CREATE TEMP TABLE

Creates an in-memory table backed by `InMemoryTableProvider`:

```sql
CREATE TEMP TABLE features (
    id    Int32 PRIMARY KEY,
    score Float32
)
```

`CREATE TEMP TABLE` works in any catalog (no `.datum-catalog.json` required). The table dies when the catalog is disposed. Temp tables support all the same column modifiers as persistent tables, including `IDENTITY`, `DEFAULT`, and `PRIMARY KEY` â€” but PK uniqueness uses the in-memory `HashSet` path (no `.datum-pkindex` sidecar).

### DROP TABLE

Removes a table from the catalog and deletes its backing files:

```sql
DROP TABLE features
DROP TABLE IF EXISTS features
```

For a persistent table this deletes the `.datum` file plus all companion sidecars (`.datum-blob`, `.datum-index`, `.datum-manifest`, `.datum-pkindex`).

### INSERT INTO

Appends rows to an existing table:

```sql
-- Literal values
INSERT INTO features VALUES (1, 'Alice', 0.95), (2, 'Bob', 0.42)

-- Column list (omitted columns get DEFAULT, else NULL, else error)
INSERT INTO features (customer_id, label) VALUES (3, 'churn')

-- From a query
INSERT INTO features (id, name, score)
SELECT id, name, score FROM raw_data WHERE score IS NOT NULL
```

#### Omitted-column resolution

For each target column not supplied by the `VALUES` row or `SELECT` projection:

1. If the column is `IDENTITY`, the catalog reserves the next value.
2. Otherwise if the column has a `DEFAULT`, the expression is evaluated against an empty frame and the result is coerced to the column kind.
3. Otherwise if the column is computed (`AS (expr)`), it's filled by a second pass after every other slot resolves â€” the expression evaluates against the partially-built row, so it can reference `DEFAULT`-filled and `IDENTITY`-filled columns.
4. Otherwise if the column is `Nullable`, `NULL` is written.
5. Otherwise the entire batch is rejected with an error naming the column.

#### Type coercion

Source values are coerced to the target column's `DataKind` losslessly:

- Integer widening (Int8 â†’ Int32) accepted; narrowing rejected at runtime when the value doesn't fit.
- Negative literals into unsigned columns rejected.
- `Float64 â†’ Float32` accepted only when the round-trip is exact (`0.1 â†’ Float32` rejects).
- String â†’ numeric / numeric â†’ string and other cross-family coercions rejected with a clear error.

#### IDENTITY

When a column is declared `GENERATED ALWAYS AS IDENTITY` (or its T-SQL alias, bare `IDENTITY`), supplying it in the `INSERT` column list is rejected. The catalog auto-fills it from the prologue's running counter; the counter is updated atomically with the data commit, so values are never reused even across crashes.

When a column is declared `GENERATED BY DEFAULT AS IDENTITY`, the column accepts user-supplied values on `INSERT` and only auto-fills from the counter when omitted. The counter is not advanced by explicit inserts:

```sql
CREATE TABLE t (id Int64 GENERATED BY DEFAULT AS IDENTITY, name String);

INSERT INTO t (id, name) VALUES (42, 'explicit');  -- explicit id, counter still at 1
INSERT INTO t (name) VALUES ('auto');              -- counter generates id=1
-- (next auto-insert would generate id=2; if you'd explicitly inserted id=2 above,
--  this would collide on PK if 'id' were a PRIMARY KEY)
```

#### PRIMARY KEY

Each `INSERT` validates that no row duplicates an existing PK or another row in the same batch. The first violation throws `PrimaryKeyViolationException` and the entire batch is aborted (no partial commit). NULL in any PK column is also rejected.

#### Computed columns

A column declared `AS (expr)` is a `GENERATED ALWAYS AS` computed column. Its value is materialised at `INSERT` time by evaluating the expression against the row being built; explicit values are rejected.

```sql
CREATE TABLE orders (
    qty Int32,
    price Float64,
    total Float64 AS (price * qty)
);

INSERT INTO orders (qty, price) VALUES (3, 19.99);
-- row reads back as (3, 19.99, 59.97)

INSERT INTO orders (qty, price, total) VALUES (3, 19.99, 100.0);
-- ERROR: column 'total' is GENERATED ALWAYS AS (computed). Drop it from the
--        INSERT column list and the catalog will compute the value.
```

The expression can reference any other non-computed column in the same row. It can also reference `IDENTITY` and `DEFAULT`-filled columns â€” those resolve before computed columns evaluate, so the expression sees the post-fill values:

```sql
CREATE TABLE conversations (
    id Int64 IDENTITY,
    title String,
    slug String AS ('conv-' || cast(id as string))
);

INSERT INTO conversations (title) VALUES ('Chat');
-- row reads back as (1, 'Chat', 'conv-1')
```

`UPDATE` rejects `SET` on a computed column. To change the column's value, change one of its referenced source columns.

```sql
UPDATE orders SET total = 0 WHERE id = 5;
-- ERROR: column 'total' is GENERATED ALWAYS AS (computed). Computed columns
--        derive their value from other columns and cannot be assigned directly.
```

##### UPDATE recomputes dependents

When an `UPDATE` touches a column that a computed column references, the computed column's value is recomputed for each affected row. For `total Float64 AS (price * qty)`, `UPDATE orders SET price = 10 WHERE id = 5` recomputes `total` for row 5 against the new `price`. `UPDATE â€¦ FROM <source>` follows the same rule â€” after every source row's `SET` has been evaluated (last-match-wins), dependents recompute against the final per-target-row state.

##### ALTER TABLE ADD COLUMN AS (...) backfills historical rows

`ALTER TABLE t ADD COLUMN sum Int32 AS (a + b)` evaluates the expression against every existing row at `ALTER` time and writes the result. Non-deterministic calls (`now()`, `uuidv4()`, `random()`) freeze their value at `ALTER` time for historical rows; subsequent `INSERT`s continue to evaluate per row. Rows whose expression resolves to NULL (e.g. `a + b` with a NULL input) stay NULL â€” they're written by the standard pump and not redundantly rewritten.

##### Computed-to-computed references are rejected

A computed column whose expression references another computed column is rejected at `CREATE TABLE` and `ALTER TABLE ADD COLUMN` time. Inline the inner expression instead:

```sql
-- Rejected at CREATE TABLE time.
CREATE TABLE t (a Int32, b Int32 AS (a + 1), c Int32 AS (b * 2))
-- Error: Column 'c': GENERATED expressions cannot reference other GENERATED columns
--        (references 'b'). Inline the inner expression instead.

-- Write it like this.
CREATE TABLE t (a Int32, b Int32 AS (a + 1), c Int32 AS ((a + 1) * 2))
```

##### Remaining limitations

- **`VIRTUAL` is not supported.** Only `STORED` (the default) â€” computed values are materialised on disk. There is no scan-time computation mode. The `STORED` keyword is implied by the `AS (expr)` form; writing `AS (expr) VIRTUAL` fails to parse.

#### RETURNING

`INSERT` accepts an optional `RETURNING` clause that surfaces a projection of the fully-resolved (post-`DEFAULT`, post-`IDENTITY`, post-computed) inserted rows. Same syntax as a `SELECT` projection list â€” column references, computed expressions with aliases, `*`, and table-qualified `t.*` are all accepted.

```sql
-- Surface the auto-generated id alongside the insert.
INSERT INTO conversations (workspace, title) VALUES ('default', 'Chat')
RETURNING id, workspace, title

-- All resolved columns, including DEFAULT-filled ones.
INSERT INTO uploads (mime, size_bytes) VALUES ('image/png', 12480)
RETURNING *

-- Computed expression with alias.
INSERT INTO orders (qty, unit_price) VALUES (3, 19.99)
RETURNING qty * unit_price AS total

-- INSERT â€¦ SELECT â€¦ RETURNING streams resolved rows for every inserted row.
INSERT INTO archive (id, payload)
SELECT id, payload FROM staging WHERE ready = true
RETURNING id
```

The `RETURNING` rows are visible only after the implicit commit succeeds â€” an `INSERT` that aborts mid-write yields no rows. Expressions evaluate against each inserted row in target-table scope; subqueries, aggregates, window functions, and references to other tables are rejected.

`RETURNING` also makes a data-modifying `INSERT` usable as a CTE body, enabling single-statement chains:

```sql
-- Insert a parent, then insert children that reference the parent's id.
WITH new_conv AS (
    INSERT INTO conversations (workspace, title) VALUES ('default', 'Chat')
    RETURNING id
)
INSERT INTO messages (conversation_id, body)
SELECT id, 'Hello' FROM new_conv
```

A CTE body that's an `INSERT` must include a `RETURNING` clause â€” without it the CTE has no rows to project.

### DELETE

Soft-deletes rows matching a predicate:

```sql
DELETE FROM features WHERE score IS NULL
DELETE FROM features                       -- unconditional: removes every live row
```

Tombstoned rows are excluded from subsequent queries. Storage is not reclaimed â€” a future compaction pass rewrites the file. Row indices passed to the provider's `DeleteRows` are linear over the live row sequence; `DELETE` walks a fresh scan and accumulates indices, so tombstones from prior `DELETE` calls don't shift the numbering incorrectly.

### ALTER TABLE ADD COLUMN

Adds a nullable column. The new column must be `NULL`able:

```sql
ALTER TABLE features ADD COLUMN risk_tier String
ALTER TABLE features ADD COLUMN risk_tier String NULL
ALTER TABLE features ADD [COLUMN] risk_tier String   -- COLUMN keyword optional
```

The column kind is **required** between the column name and any modifier clause (`AS`, `DEFAULT`, `IDENTITY`, `GENERATED`). DatumIngest does not infer the column kind from a `GENERATED AS (expr)` body â€” match PostgreSQL by specifying the kind explicitly:

```sql
-- âťŚ Rejected: missing column kind.
ALTER TABLE y ADD COLUMN hash AS (sha256(content))

-- âś… Valid: sha256 returns UInt8[], so declare that.
ALTER TABLE y ADD COLUMN hash UInt8[] AS (sha256(content))
```

`DEFAULT`, `GENERATED ALWAYS AS (expr)` (computed), and `GENERATED [ALWAYS|BY DEFAULT] AS IDENTITY` are all accepted on `ADD COLUMN`:

- `ADD COLUMN col Kind DEFAULT <expr>` â€” existing rows read `NULL` for the new column; subsequent `INSERT`s that omit the column auto-fill via the per-row evaluator (`DEFAULT now()` on `ADD COLUMN` gives every future `INSERT` a fresh timestamp).
- `ADD COLUMN col Kind AS (expr)` â€” historical rows are backfilled by scanning the post-mutation table, evaluating the expression per row, and dispatching one page-COW `UPDATE` pass that installs the computed values. Subsequent `INSERT`s evaluate the expression per row as usual.
- `ADD COLUMN col Kind GENERATED ALWAYS AS IDENTITY [(seed, step)]` â€” historical rows are backfilled with sequential counter values starting at `seed`; the running counter advances past the backfill so subsequent `INSERT`s continue cleanly. Rejected when the table already has an IDENTITY column.
- `ADD COLUMN col Kind GENERATED BY DEFAULT AS IDENTITY [(seed, step)]` â€” same backfill, but subsequent `INSERT`s may supply explicit values (see the IDENTITY section above for `BY DEFAULT` semantics).

`NOT NULL` on `ADD COLUMN` is rejected unless paired with a `DEFAULT` or `AS (expr)` â€” there's no way to materialise a non-null value for existing rows without a user-supplied backfill expression.

### ALTER TABLE DROP COLUMN

Soft-drops a column by name:

```sql
ALTER TABLE features DROP COLUMN risk_tier
ALTER TABLE features DROP COLUMN IF EXISTS risk_tier
```

The column block stays in the footer (marked `Tombstoned`) for compaction-time reclamation, but is hidden from `GetSchema()` and from subsequent scans. Dropping a column that's part of the table's `PRIMARY KEY` is rejected.

**The column name is freed immediately on `DROP`.** A subsequent `ALTER TABLE ... ADD COLUMN <same-name> <kind>` succeeds and gets a fresh storage slot:

```sql
CREATE TABLE y (id Int32, hash String);
-- ... oops, wanted UInt8[] for hash, not String.
ALTER TABLE y DROP COLUMN hash;
ALTER TABLE y ADD COLUMN hash UInt8[];   -- reuses the name; existing rows read NULL for the new hash
```

Matches PostgreSQL â€” once a column is dropped, the name is reusable. The tombstoned footer entry lingers until compaction; it remains addressable by the storage layer for that purpose but is invisible at the SQL surface. Double-`DROP` of the same name without an intervening `ADD` errors (unless you use `IF EXISTS`).

### Batch execution

Multiple statements can be combined into a single batch. Statements execute sequentially; on failure, execution stops and no further statements run.

```sql
DROP TABLE IF EXISTS features;
CREATE TABLE features (
    id    Int32 PRIMARY KEY IDENTITY,
    name  String NOT NULL,
    score Float32 DEFAULT 0.0
);
INSERT INTO features (name, score) VALUES ('alice', 0.95), ('bob', 0.42);
INSERT INTO features (name) SELECT name FROM raw_data WHERE score IS NOT NULL;
DELETE FROM features WHERE score IS NULL
```

Semicolons between statements are optional â€” statement parsers are keyword-anchored, so consecutive statements without a separator disambiguate cleanly. Trailing semicolons are silently ignored.

For procedural batches â€” variables, conditionals, loops, multi-variable `SELECT` assignment â€” see [Procedural Statements](procedural.md).
